# Bridge 11 — IEC 61850 (power grid / substation automation)

> **Prerequisite:** `00-FRAMEWORK.md`.

**Priority:** medium with high strategic value. **Safety ceiling:** **READ-ONLY** initially. Write capabilities (operate breakers, change tap positions) are deliberately not implemented in this bridge — they belong on certified RTUs and DCS-class systems with their own SIS, not on a Java bridge.

## 1. Domain context

[IEC 61850](https://webstore.iec.ch/publication/6028) is the leading interoperability standard for electrical substations and power utility automation. It defines:

* A **data model** — `LDeviceA/MMXU1.PhV.phsA.cVal.mag.f` is the kind of path you read in this protocol. Logical Devices, Logical Nodes, Data Objects, Data Attributes — five levels deep.
* **Information services** — MMS for client/server, GOOSE for fast event multicast, Sampled Values for synchrophasor-rate streams.
* **Engineering exchange** — SCL (Substation Configuration Language) XML files describe the entire substation configuration.

Substations expose **measurements** (currents, voltages, frequencies, power factors), **status** (breaker open/closed, isolator position), **events** (protection trips, alarms), and **control** (operate, select-before-operate, cancel). For Jneopallium we read the first three. Control is not in scope.

## 2. Maven dependency

```xml
<!-- libIEC61850 has Java bindings via JNI; the libiec61850 project also
     ships a pure-Java client. The pure-Java route avoids native libs but
     covers fewer edge cases. -->
<dependency>
    <groupId>org.openmuc</groupId>
    <artifactId>iec61850bean</artifactId>
    <version>1.7.0</version>
</dependency>
```

`iec61850bean` (OpenMUC) is the established pure-Java MMS client. Verify the latest version before merge.

## 3. Why read-only

* IEC 61850 control writes use the select-before-operate (SBO) pattern with operator confirmation — there's no substitute for the human operator path. Trying to wrap SBO in a JVM-resident bridge undermines the protocol's safety model.
* Substation control is a SIL-classified function (typically SIL 2/3) that lives on certified RTUs and protection relays. A Jneopallium bridge writing to a `Oper` data attribute would be acting outside its certification scope.
* Reading is sufficient to add value: state estimation cross-checks, anomaly detection, equipment-degradation modelling, demand response advisory.

A bridge that opens a write surface here is a different bridge — name it `iec61850-control` and certify it separately. Do not extend this one.

## 5. Signal mapping

| IEC 61850 source | Jneopallium signal | Notes |
|---|---|---|
| `MMXU` (measurement, e.g. PhV/PhA/W/Hz) | `MeasurementSignal` | Quality from `q` attribute (GOOD/INVALID/QUESTIONABLE/SUBSTITUTED → mapped to `Quality.GOOD/BAD/UNCERTAIN/UNCERTAIN`). |
| `XCBR.Pos` (breaker position) | `InterlockSignal` (state-only, never tripped from this side) | Status, not a directive. |
| `PIOC`/`PTOC` operate signals | `AlarmSignal(CRITICAL, "PROTECTION_OPERATE")` | These are the protection trips — read-only awareness, not commands. |
| `LLN0.Mod` (logical node mode) | bridge metadata | |
| GOOSE events | `AlarmSignal` or `InterlockSignal` depending on type | GOOSE multicast requires a separate listener thread per VLAN. |

## 6. Configuration

```yaml
ied:
  - id: "RELAY-A1"
    host: "10.50.1.10"
    port: 102                          # MMS standard port
    sclFile: "/etc/jneopallium/scl/SUB1.icd"
    reportControlBlock: "LD0/LLN0.RP.urcbA01"

reads:
  # Direct DA path
  - bindingId: "BUSBAR-V"
    iedId: "RELAY-A1"
    daPath: "LD0/MMXU1.PhV.phsA.cVal.mag.f"
    signalTag: "SUB1.BUSBAR.VA"

  - bindingId: "FEEDER-1-CURRENT"
    iedId: "RELAY-A1"
    daPath: "LD0/MMXU2.A.phsA.cVal.mag.f"
    signalTag: "SUB1.FEED1.IA"

  - bindingId: "BREAKER-CB1-POS"
    iedId: "RELAY-A1"
    daPath: "LD0/XCBR1.Pos.stVal"
    signalTag: "SUB1.CB1.POS"

events:
  # Subscribe to all PROT_OP indications via report
  - bindingId: "PROT-OPS"
    iedId: "RELAY-A1"
    reportControlBlock: "LD0/LLN0.RP.urcbProt01"
    targetSignal: "ALARM"
    severityMap:
      "PIOC": "CRITICAL"
      "PTOC": "CRITICAL"
      "PTUV": "HIGH"

audit:
  localAuditFile: "/var/log/jneopallium/iec61850-audit.jsonl"
# `writes:` block is rejected at config-load with a clear message.
```

The bridge **requires** an SCL file per IED. SCL provides the ground-truth data model; without it, DA paths cannot be reliably resolved. The bridge fails fast at startup if SCL is missing or malformed.

## 8. Phase plan

| Phase | Goal |
|-------|------|
| 1 | MMS client against an open simulator (`libiec61850` ships a sample server). Read MMXU and XCBR. |
| 2 | Report Control Block (RP/BR) subscriptions for buffered/unbuffered reports. Map to `AlarmSignal`. |
| 3 | GOOSE listener (separate thread, requires raw socket / VLAN config). Optional based on deployment. |
| 4 | **Permanently disabled.** Control surface is a separate bridge. |

## 9. Bridge-specific scenarios

| # | Scenario | Setup | Expected |
|---|----------|-------|----------|
| **S7** | Sample server | Run `libiec61850`'s `server_example_basic_io` on `localhost:102` | Bridge connects, MMXU reads emit `MeasurementSignal` |
| **S8** | Quality propagation | Server flagged `q.invalid` for one DA | Emitted signal carries `Quality.BAD`; value passed through |
| **S9** | SCL missing | YAML references `sclFile` that doesn't exist | Bridge fails at startup with the missing file path |
| **S10** | Report subscription | Server triggers a buffered report | Bridge emits `AlarmSignal` per the report contents |
| **S11** | No write surface | Compile-time check: `Iec61850CommandOutputAggregator` does not exist | Build verifies absence |

## 10. Risks

| # | Risk | Mitigation |
|---|------|------------|
| R1 | Misuse: someone tries to add control writes | Architecture review item: write surface is deliberately absent; PRs adding it must include a separate certification plan. |
| R2 | GOOSE on the wrong VLAN spamming the bridge | Per-binding GOOSE filter (appId, source MAC); overflow drops with audit. |
| R3 | SCL dialect drift between vendors | OpenMUC's SCL parser handles common cases; document tested vendor stacks (ABB, Schneider, SEL). |
| R4 | Time sync (PTP/IRIG-B) for sampled-values use cases | Out of scope for v1; Sampled Values support is deferred. Use cases requiring it are documented as future work. |
